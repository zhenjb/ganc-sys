### STATE-02 — initial local state

`internal/state/local_state.go::NewLocalState()`:

- Creates an empty `AccountState` and computes `rootA = ComputeRoot(empty)`.
- Initial root is **derived**, not hard-coded — so P1 genesis (ONCHAIN-03) can call the same code path to seed `currentStateRoot` and never drift.
- Concurrency: all mutations go through a `sync.Mutex`; readers (`Root`, `Account`, `Snapshot`) take the read lock.

### STATE-03 — apply deposit

`internal/state/local_state.go::ApplyDeposit(DepositRecord) (newRoot, error)`:

1. Validates `depositId / owner / denom / amount` (amount must parse as non-negative integer and be `> 0`).
2. Idempotency-guards on `depositId`. Replaying the same on-chain `DepositQueued` event from the indexer **must not** double-credit. Returns `ErrDepositAlreadyApplied` on replay.
3. Credits `Balance` via `big.Int` addition. `Nonce` is **not** touched on deposit (changes only on withdrawal — STATE-05).
4. Recomputes root deterministically from the new account snapshot and returns it.

### Root encoding (MVP, subject to ZK-02 re-lock)

```
sha256( "zkdex/state/v1" || '|' || canonicalJSON(accounts sorted by (owner, denom)) )
```

- Hex output, `0x`-prefixed.
- Order-independent: same final state → same root, regardless of deposit ordering. Asserted by `TestComputeRoot_OrderIndependent`.
- **Why a state-snapshot hash (not a transition hash):** the verifier in P1 ONCHAIN-08 must derive the public input `newStateRoot` from the post-state, not from a chain of historical transitions. Snapshot-hash keeps deposit + withdraw symmetric in STATE-05 without re-coding.
- **Flagged for P2 (ZK-02):** if the circuit needs Poseidon / a field-friendly hash, replace `pkg/hash` and re-run `gen_state_vectors`. Public-input *order* and *naming* don't change; only the digest.

## Tests

`internal/state/local_state_test.go` covers:

- Initial root is deterministic across `NewLocalState()` calls and `0x`-prefixed.
- Apply once: balance 100, nonce unchanged, root advances.
- Apply same `depositId` twice → `ErrDepositAlreadyApplied`, balance stays 100.
- Apply two distinct deposit IDs to same account → balance accumulates (100 + 40 = 140).
- All invalid `DepositRecord` shapes (empty id/owner/denom, zero amount, negative, non-numeric) → `ErrInvalidDepositRecord`.
- Order independence: applying `{d1,d2}` vs `{d2,d1}` yields the same final root.
